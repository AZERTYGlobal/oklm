OKLM

Open Keyboard Layout Model — one open, human-maintainable source of truth per keyboard layout, exported everywhere it is needed.

Draft 0.1 — schemas, example and validator available

Keyboard layouts are still distributed as OS-specific files, static images, installer packages and informal documentation. OKLM defines a common representation so a layout can be described once — keys, levels, dead keys, legends, metadata — then exported to LDML Keyboard, operating-system formats, web testers, keycap maps, dynamic key displays and assistant knowledge files.

GitHub repository Specification JSON Schemas

Design principles

  • Human-maintainable: an .oklm.json manifest must be readable, correctable and versionable without a proprietary generator.
  • Standards-aligned: ISO/IEC 9995 key numbers and (group, level) terminology; the text core compiles to Unicode LDML Keyboard 3.0 rather than reinventing it.
  • Validated: JSON Schemas plus a CLI validator for manifests and conversion reports.
  • Loss-aware conversions: every OKLM ⇄ LDML conversion produces a machine-readable report — nothing is silently discarded.

A key, in five lines

{
  "id": "D01", "hid": "0x14", "xkb": "AD01", "code": "KeyQ",
  "levels": { "1": "a", "2": "A" },
  "categories": ["letter", "french"]
}

Status

OKLM is a public draft, not yet a published standard. The manifest schema (draft 0.1), a conversion-report schema, a minimal AZERTY Global example and a reference validator are available in the repository. Reference layouts: AZERTY Global and QWERTY Global.

# Keyboard input from switch to symbol

## Scope and terminology

For an engineer designing an open, OS-independent keyboard-layout model, the crucial distinction is between **physical position**, **host key identity**, and **produced output**. A physical position is the switch location on the keyboard matrix or grid. A host key identity is the code that an operating system uses before layout translation, such as a Windows scan code and `VK_*`, a macOS virtual key code, or a Linux `EV_KEY`/XKB keycode. The produced output is the result after layout logic, modifier state, dead-key state, and composition have all been applied: that result may be a Unicode string, a keysym, or a non-text action such as `Return`, cursor movement, or a shortcut dispatch. HID itself is explicit that keyboard usages are for implementing keyboards and should use the “closest equivalent key position” across languages, which is why the same physical key can be “Y” on US keyboards and “Z” on German keyboards while keeping the same HID usage. citeturn19view1turn19view2turn23view0turn36view0

The most compact glossary for the code systems involved is below.

| Term | Precise definition |
| --- | --- |
| **HID Usage ID** | An identifier within a HID usage page that names the purpose of a control. On keyboard page `0x07`, usages are primarily key **positions/functions**, not characters. citeturn19view1turn19view2 |
| **Scancode** | A low-level device or transport code for a key event. In Win32 documentation, it is a device-dependent identifier for the key; in Windows messages the delivered value is the **Scan Code Set 1 make code**. citeturn23view0turn20view0turn42view0 |
| **Virtual key** | A device-independent OS API code that identifies the purpose of a key, such as Windows `VK_*`; on macOS, Apple likewise documents virtual key codes used in keyboard events. citeturn23view0turn22view0turn24search3 |
| **Keycode** | A host-side numeric code representing a physical key before symbol translation. On Linux `evdev`, it is the `EV_KEY` code; in XKB, `xkb_keycode_t` is “a number used to represent a physical key on a keyboard.” citeturn9search0turn35search6turn36view0 |
| **Keysym** | In X11/XKB, a numeric symbol identifier bound to a key level. It may denote a character, dead key, modifier, or system action. citeturn10search1turn36view0 |
| **Dead key** | A key that initially produces no final character, but modifies the following keystroke or keystrokes to compose another result. citeturn20view0turn46view0turn24search0 |

## End-to-end pipeline

The full pipeline is easiest to understand if you split it into **electrical detection**, **transport encoding**, **OS normalization**, and **layout translation**.

```text
Physical switch closes
→ keyboard matrix row/column is sensed
→ keyboard firmware debounces and identifies a physical switch position
→ firmware encodes a transport-level key event
   • usually USB HID Keyboard/Keypad usage(s) on page 0x07
   • or, on legacy PS/2 paths, scan-code-set bytes
→ host transport/driver stack receives the report or byte stream
→ OS keyboard driver normalizes it into platform-native host codes
   • Windows: scan code → virtual key → message / Unicode translation
   • macOS: HID event → virtual key code → keyboard-layout translation
   • Linux: scancode → kernel keycode / evdev EV_KEY → XKB/X11/Wayland translation
→ layout engine applies modifier state, dead-key state, compose rules, and locale rules
→ application receives either character(s), keysym(s), or non-text key actions
```

At the **hardware and firmware** layer, the keyboard scans a switch matrix and decides that a particular physical switch has changed state. That physical fact is still pre-layout: it is not yet a letter or symbol. In the USB HID model, the device then reports keyboard usages from page `0x07`, and the HID Usage Tables explicitly say that this page is for the key codes used to implement a USB keyboard. The same document also says vendors should keep the same usage for the “closest equivalent key position” even when keycaps change with language, which is the formal reason HID keyboard usages are fundamentally **position-oriented** rather than character-oriented. citeturn19view1turn19view2

On a **legacy PS/2 path**, the transport encoding is different: the keyboard sends scancode-set bytes instead of HID usages. Microsoft’s keyboard scan code specification states that PS/2-compatible keyboards must support **Scan Set 2**, that Windows systems expect **Scan Set 1** values in the rest of the OS, and that on Microsoft operating systems the keyboard sends Set 2 on the wire while the i8042 translation path converts that to Set 1. The same Microsoft specification says Scan Set 3 is not used or required for Windows PS/2 operation. That is a legacy host-transport compatibility path, not a language layout. citeturn42view0

On **Windows**, the keyboard device driver receives scan codes, maps them to **virtual-key codes**, and then the selected keyboard layout translates those into keystroke and character messages. Microsoft’s overview is explicit about ownership: the device driver provides device-independent keyboard support by mapping scan codes to VKs, while the selected keyboard layout provides language-dependent translation. `TranslateMessage` and `ToUnicodeEx` are the layers that turn `VK + keyboard state (+ scan code)` into character output, including dead keys. The same documentation notes that applications can use `MapVirtualKeyW(..., MAPVK_VSC_TO_VK_EX)` to distinguish left and right Shift/Ctrl/Alt when the base virtual key is only `VK_SHIFT`, `VK_CONTROL`, or `VK_MENU`. citeturn23view0turn20view0turn8search7

On **macOS**, Apple’s public documentation exposes the same split, though with less complete published tabulation. Current Apple docs define `CGKeyCode` as the virtual key code family used for low-level keyboard events, and document that the system normally translates the key code in a keyboard event into a Unicode string based on the keyboard ID in the event source. Apple’s `UCKeyboardLayout` documentation says the Unicode keyboard-layout resource contains the data needed to map virtual key codes to Unicode characters, and `UCKeyTranslate` explicitly takes a **virtual key code**, **modifier state**, and **dead-key state** and returns Unicode output. Historically, Apple’s Event Manager documentation showed this as `raw key code → 'KMAP' → optional 'itlk' remap → 'KCHR' → character code`; the ownership model is the same even though the data structures have modernized. citeturn24search3turn24search6turn24search13turn24search0turn33view0turn32view3

On **Linux**, the kernel input stack sits between transport and layout. Kernel input documentation describes a driver-maintained keycode table that maps device scancodes to input-system keycodes, and `evdev` is documented as the preferred userspace interface that exposes architecture-independent event codes. Above that, X11 and Wayland commonly use **XKB**. In libxkbcommon, a keymap maps physical keycodes to symbols and actions; `xkb_state_key_get_one_sym()` returns a keysym, and `xkb_state_key_get_utf8()` returns the resulting Unicode string. The key point is that Linux kernel input and XKB are two distinct layers: `evdev` gives you physical-ish keycodes, while XKB owns the layout logic. citeturn9search2turn35search6turn9search0turn36view0turn10search8

The important practical conclusion is that **layout logic belongs to the host layout engine, not to HID or PS/2 transport codes**. HID usages and scancodes tell the OS *which switch position or function* changed; the OS layout layer decides whether that means `q`, `Q`, `á`, `@`, `Return`, or a shortcut/action. The exception is firmware-side remapping: if firmware deliberately emits a different HID usage than the switch’s conventional position, then the host will only ever see the remapped usage. That is an inference from the HID and driver model, and it matters a great deal for any portable format. citeturn19view2turn23view0turn24search13turn36view0

## Code systems in the major desktop OSes

The major code systems are not interchangeable; each solves a different problem.

**USB HID Usage IDs** on page `0x07` identify keyboard controls in a device report. They are not characters. HID 1.4 says the keyboard page is “for key codes to be used in implementing a USB keyboard,” and its language note instructs vendors to keep the same usage for the closest equivalent physical position across languages. That is why the QWERTY `Q` position keeps the same usage even when a local layout prints `A` or `Q` or `Й` on that keycap. citeturn19view1turn19view2

**PS/2 scan code sets** are transport byte encodings for key press and release events. Microsoft’s scan code specification distinguishes Set 1, Set 2, and Set 3; requires PS/2 keyboards to support Set 2; and explains why Windows applications see Set 1 values on the legacy PC path. A “scancode” in this sense is a transport- or driver-near key-event code, not a symbol. citeturn42view0

**Windows virtual-key codes (`VK_*`)** are device-independent API identifiers. The driver maps scan codes to VKs; then the keyboard layout uses VK plus keyboard state to determine output. For letter and digit keys, the VK set uses stable symbolic identities such as `VK_Q`, `VK_A`, and `VK_SPACE`; for punctuation and locale-sensitive positions it uses `VK_OEM_*` families such as `VK_OEM_3` and `VK_OEM_102`. In message handling, the scan code and extended flag are both carried in `lParam`, and left/right differentiation for Shift/Ctrl/Alt can be recovered with `MapVirtualKeyW`. citeturn23view0turn22view0

**macOS virtual key codes** are Apple’s host-side positional codes for keyboard events. Apple’s current APIs expose them as `CGKeyCode` / `NSEvent.keyCode`, and `UCKeyTranslate` maps them to Unicode using the selected input source. Apple’s historical Event Manager diagrams are especially useful for systems work because they show that the same family of virtual key codes sits between raw key identity and character translation. The difficult part for portability is that Apple’s published positional diagrams are geometry-sensitive: the same code family is reused across ANSI and ISO hardware variants, and some representative codes vary by geometry. citeturn24search3turn24search13turn24search0turn33view1turn34view0

**Linux kernel keycodes and `evdev` keycodes** are the host-side key identities exposed by `/dev/input`. Kernel docs describe the driver’s scancode→keycode table and `EV_KEY` event model, while `evdev` provides the standard userspace interface. Those keycodes are still pre-layout. **X11 keycodes** are then another namespace used by the X server/XKB; in current X11-compatible keymaps there is typically a historical **offset of 8** between evdev-style raw keycodes and X11 wire keycodes. **Keysyms** are yet another namespace, representing the symbol/action bound at a given key level. citeturn9search2turn35search6turn11search0turn36view0

The “typical X11 keysym” cells assume a standard Latin layout and therefore illustrate the *output symbol usually bound at level 1*, not an invariant identity. The whole point is that the physical key and the character are often different things. citeturn21view0turn19view2turn33view1turn34view0turn35search6turn11search0turn45view0

## Modifiers, levels, and composition

The biggest structural difference between operating systems is not the base key identity, but **how modifier state and shift levels are modeled**.

In **Windows**, the public API model is relatively imperative. The keyboard layout examines a message’s virtual-key code and keyboard state, taking into account keys such as Shift and Caps Lock, and then produces character messages. `ToUnicodeEx` likewise translates `(VK, scan code, key state, keyboard layout)` into Unicode and documents that a previously entered dead key may still be stored in the keyboard-layout state. Windows exposes modifier keys as virtual keys such as `VK_SHIFT`, `VK_CONTROL`, `VK_MENU`, `VK_LSHIFT`, `VK_RSHIFT`, `VK_LMENU`, and `VK_RMENU`, but it does not expose an XKB-style first-class “LevelThree” model in the Win32 API. Instead, alternate levels are layout-defined shift states. Microsoft’s own keyboard-layout documentation for modern French AZERTY explicitly shows an `AltGr` layer, which demonstrates that Windows layouts do model an alternate shift state even though the low-level key is still just right Alt / `VK_RMENU`. citeturn23view0turn8search7turn40search3turn22view0turn40search12

In **XKB**, by contrast, modifiers and levels are explicit data-model elements. XKB distinguishes **real modifiers** (`Shift`, `Lock`, `Control`, `Mod1`…`Mod5`) from **virtual modifiers** such as `Alt`, `NumLock`, `LevelThree`, and `LevelFive`. It also models three modifier states: **depressed**, **latched**, and **locked**. A key’s **type** defines which modifiers are relevant and how those modifiers select a **shift level**. In standard current configurations documented by libxkbcommon, `LevelThree` is usually associated with ISO level 3 / AltGr and `LevelFive` with ISO level 5, but those mappings are conventions of the active keymap, not universal laws. XKB also makes locale-sensitive Caps behaviour explicit: alphabetic key types typically let `Lock` select the upper-case level for letters while leaving number-row keys unaffected. citeturn11search0turn13view0turn13view1

The old **core X11** model used a simpler but related idea: `MODE SWITCH` selected Group 2 keysyms when attached to one of `Mod1` through `Mod5`, `Caps Lock` and `Shift Lock` were behaviours attached to the Lock modifier, and the first four keysyms on a keycode were interpreted as Group 1/Group 2 with and without Shift. That is where X11’s older `Mode_switch` concept comes from. Modern XKB generalizes that model into groups, types, and virtual modifiers such as `ISO_Level3_Shift` and `ISO_Level5_Shift`. citeturn14search1turn14search2turn12search0

On **macOS**, Apple’s public documentation is less explicit about levels as named objects, but the same functional structure exists. `UCKeyTranslate` takes **virtual key code**, **modifier key state**, and **dead-key state** and returns Unicode output. `NSEvent.characters` is documented as being derived from a keyboard mapping that associates key combinations with Unicode characters, and `characters(byApplyingModifiers:)` shows that modifier application is a property of the layout translation layer rather than of the transport code. In other words, macOS has levels, but Apple documents them through the translation API and layout-resource model rather than through XKB-style named selectors. citeturn24search0turn24search13turn30search1turn30search9

Dead keys and composition happen at different layers on different systems. On **Windows**, `TranslateMessage` may emit `WM_DEADCHAR` or `WM_SYSDEADCHAR`, and `ToUnicodeEx` explicitly documents a stored previous dead key in the keyboard-layout state. If the next keystroke can combine, the final combined character is emitted; if it cannot, Windows may emit the dead accent separately plus the following character. Microsoft also warns that `ToUnicode` changes kernel-mode keyboard-buffer state, so text translation is stateful. citeturn20view0turn8search7turn40search3

On **XKB/libxkbcommon**, a dead key is just a keysym such as `dead_acute`, and broader composition is handled through a **Compose table** and a **compose-state machine**. The compose docs show both `<dead_acute> <a> -> "á"` and `<Multi_key> <A> <T> -> "@"`, explain that compose files are locale-dependent, and define state transitions such as `COMPOSING`, `COMPOSED`, and `CANCELLED`. They also note that cancellation behaviour is not universal: libX11 typically swallows the cancelling keysym, whereas Windows approximately replays the underlying dead-key output. That difference matters directly for any portable format that attempts to model chained dead keys or Compose sequences. citeturn46view0

On **macOS**, Apple’s public APIs expose dead-key state directly in `UCKeyTranslate`, which is strong evidence that dead-key processing is part of the keyboard-layout translation layer. Apple’s retrieved public docs do not, however, expose an X11-style general Compose-key mechanism as a similarly explicit public primitive; broader composition on macOS is handled by input sources and text input services rather than by a documented `Compose` analogue in the sources reviewed here. citeturn24search0turn24search13

## Geometries and physical naming

Any open layout model must treat **geometry** as first-class data. XKB’s own configuration model says keyboard models depend on whether the hardware is **ANSI, ISO, JIS, or ABNT**, and specifically notes that Japanese and Korean keyboards may have keys such as **Muhenkan** or **Hanja** that matter to input methods. That is a direct acknowledgement from the XKB ecosystem that “keyboard layout” and “keyboard geometry” are different axes. citeturn37search0turn38search0

XKB’s standard positional naming convention is extremely useful as an OS-independent abstraction. In the `xkb_keycodes` section, names like `<AE01>` and `<AD01>` denote positions in the ISO/IEC 9995 grid. Libxkbcommon explains the convention explicitly: `<AE01>` means the main alphanumeric section, row `E`, column `01`. It then gives examples such as `<AE01> = 10`, and later in the symbols section it uses `key <AD01> { [ q, Q ] };`. This is exactly the abstraction that separates **physical position** from **layout output**. citeturn11search0turn45view0

For geometry-specific extras, the transport and OS layers do not agree on one universal numbering, but they do preserve distinct identities. Microsoft’s current Windows scan-code table includes `Keyboard Non-US Backslash and Pipe` (`0x64`), `International1–5` (`0x87–0x8B`), and `LANG1–4` (`0x90–0x93`) HID usages, which are precisely the kinds of keys used for ISO, JIS, and other non-US geometries. In other words, standard host stacks already distinguish these keys; what differs is the host-native code number and the layout meaning bound to them. citeturn21view0

One subtle but important trap appears on **macOS**. Apple’s published ANSI and ISO virtual-key diagrams show that some keycodes are geometry-sensitive. In Apple’s ANSI diagram, the key left of `1` is shown as `0x32`. In Apple’s ISO diagram, the key left of `1` is shown as `0x0A`, while the extra ISO key near the left Shift area is shown as `0x32`. That means a naïve “macOS keycode as universal physical ID” approach is unsafe across geometries. This is one of the clearest concrete arguments for using a geometry-aware, OS-independent physical identifier instead of any one host’s raw numeric code set. citeturn33view1turn34view0

## Implications for an OS-independent layout model

A portable format should therefore adopt a **two-layer identifier scheme**.

The primary identifier should be a **stable physical-position ID**, not a character and not an OS numeric code. The best practical choice is an **ISO/IEC 9995 / XKB-style position namespace** such as `TLDE`, `AE01..AE13`, `AD01..AD13`, `AC01..AC12`, `AB01..AB11`, `SPCE`, `RTRN`, `BKSP`, `CAPS`, `LFSH`, `RTSH`, and the geometry-specific extras such as the ISO extra key conventionally known in XKB tooling as `<LSGT>`. This style is already designed to mean “the key in this grid position” rather than “the key that currently prints this glyph.” It maps naturally to how HID page `0x07` is supposed to be used and to how XKB already thinks about keyboard positions. citeturn19view2turn11search0turn45view0

The secondary layer should carry **interop metadata** for host stacks: USB HID usage, Windows Scan-1/VK, macOS virtual key code, Linux evdev keycode, and optional X11 wire keycode. Those should be treated as **derived platform projections**, not as the canonical identity. This is essential because the same physical key may be `HID 0x14`, `Scan-1 0x10`, `VK_Q`, `macOS 0x0C`, `evdev 16`, and XKB `<AD01>` all at once. It is also essential because some projections are geometry-sensitive or only conventional, such as macOS code reuse across ANSI and ISO boards and X11’s common `+8` offset from evdev raw keycodes. citeturn21view0turn23view0turn33view1turn34view0turn11search0

A portable layout format should then model output as a **state machine over physical keys**, not as a flat map from key to character. At minimum it should represent: physical key; logical modifiers; modifier states (**depressed**, **latched**, **locked**); one or more **levels**; optional **groups**; and optional **dead-key / compose state**. This is closest to the XKB model, but it is also the only model rich enough to describe Windows AltGr-style alternate states, Caps-sensitive alphabetic behaviour, multi-group layouts, and dead-key composition without loss. The format should keep **modifiers** separate from **selectors**: `Shift`, `Control`, and `Lock` are not the same thing as “third level” or “fifth level,” even if a specific OS/layout realizes those selectors through right Alt or another physical key. citeturn13view0turn13view1turn23view0turn24search0

Dead keys and composition need their own explicit representation. A robust model should not treat a dead key as simply “an output character that happens to be invisible”; it should model it as a **composition action** with data tables for valid follow-ups and well-defined cancellation behaviour. If the format aims to round-trip exactly across operating systems, it should make cancellation policy configurable or at least annotatable, because libX11/libxkbcommon and Windows do not document identical cancellation semantics. Compose sequences should be separable from the base keymap, much like XKB separates keysyms from Compose files. citeturn46view0

The known traps are predictable. Do not use produced characters as physical identifiers. Do not assume Windows scan codes and Linux evdev keycodes are interchangeable just because many standard PC alphanumeric keys share the same numbers. Do not assume X11 keycodes equal evdev keycodes. Do not assume one macOS virtual keycode always means one same physical position across ANSI and ISO boards. Do not assume an `AltGr` label means the same underlying modifier model on every OS. And do not assume that if firmware remapped a key before the host saw it, the host can reconstruct the original switch position. Those are exactly the failure modes that make keyboard-layout interchange formats brittle. citeturn19view2turn23view0turn11search0turn33view1turn34view0turn40search12

## Open questions and source notes

The highest-confidence facts in this report come from the USB HID Usage Tables **version 1.4** dated **2023-01-26**, Microsoft’s current Windows keyboard-input documentation updated in **2025**, Linux kernel input documentation plus current libxkbcommon documentation, and Apple’s current API semantics for `CGKeyCode`, `UCKeyboardLayout`, and `UCKeyTranslate`. citeturn18view0turn20view0turn22view0turn35search6turn36view0turn24search0turn24search13

The main limitation is **Apple positional keycode coverage**. Apple’s current docs clearly document the existence and use of macOS virtual key codes and Unicode translation, but they do not publish an easy modern full text table of all keycodes by current geometry in the sources reviewed here. The positional numbers in the cross-reference table therefore rely on Apple’s historical Inside Macintosh diagrams, which are authoritative Apple publications but old. They are excellent for representative engineering analysis and for showing the geometry trap, but if you are implementing production-grade Mac interop you should verify geometry-specific constants against the target SDK headers as a final step. citeturn24search3turn24search0turn33view1turn34view0

A second limitation is that **PS/2 scan-code-set discussion is inherently legacy-specific**. The Set 2 → i8042 → Set 1 description is accurate for the Windows-compatible PS/2 path documented by Microsoft, but modern USB keyboards go through HID and bypass that legacy translation mechanism. An OS-independent model should therefore treat PS/2 sets as one transport family among several, not as the canonical abstraction. citeturn42view0turn20view0

# Complex Script Input and the Boundary Between Layouts, Transforms, and IMEs

## Problem framing

Engineers building an open keyboard-layout model need a strict separation between three different things that are often conflated in practice. A **keyboard layout** is a keymap: it maps physical or touch keys, modifiers, and sometimes dead keys to characters. In the Unicode LDML keyboard specification, a keyboard layout consists of a base map, one or more key maps, and optional transforms; an **IME** is a separate component intended for large character sets and typically uses contextual logic and candidate UI. The same specification also states that keyboard implementations deliver **plain text** to the platform or application, which is a crucial limit for any declarative layout model. citeturn17view0turn19view1

A practical IME pipeline is usually: **keystrokes → composition or pre-edit → optional candidate or conversion UI → commit**. On Windows, TSF exposes composition objects and candidate-list UI elements, while legacy IMM32 exposed composition strings, candidate lists, and conversion status. On Android, IMEs maintain a composing region with `setComposingText()` and replace it with committed text using `commitText()`. On Linux frameworks, the same concepts appear explicitly as **preedit** and **candidate list**. Apple’s InputMethodKit similarly exposes candidate windows and per-session input controllers. citeturn11search3turn12view0turn12view1turn16search0turn16search2turn14view3turn10search0turn10search1turn10search5

That pipeline is conceptually different from a static layout. A static layout can emit characters or dead keys, and a richer declarative layout can also perform deterministic local rewrites and reorderings. What it does **not** do is maintain lexical dictionaries, rank candidates, learn from user history, segment words semantically, or perform phonetic-to-logographic conversion over a large hypothesis space. Those behaviours belong to a full IME layer, not to a portable layout description. This boundary is not just conceptual; it is reflected in OS APIs and in Unicode’s own separation between keyboard layouts and IMEs. citeturn17view0turn12view2turn15view2turn13view4turn14view3

It is also important to keep **input** separate from **rendering**. Arabic contextual joining is a rendering rule; the Unicode core specification states that Arabic, Syriac, and Mandaic shaping rules are for rendering, while Hebrew does not require shaping apart from separately encoded final forms. Likewise, right-to-left ordering is governed by the Unicode Bidirectional Algorithm, not by the keyboard layer. In other words: a keyboard chooses code points; the text stack decides how those code points are displayed. citeturn38search0turn39search0turn39search1

## Platform framework layers

On **Windows**, the current framework is **Text Services Framework**. Microsoft’s current guidance for Windows apps is explicit: IMEs should use TSF, and IMEs implemented through legacy **IMM32** are blocked for Windows app scenarios. TSF exposes composition objects (`ITfContextComposition`, `ITfComposition`) and candidate-list UI elements; legacy IMM32 exposed equivalent concepts through APIs such as `ImmGetCompositionString`, `ImmGetCandidateList`, `ImmGetConversionList`, and dictionary registration functions. That tells you exactly where the IME boundary sits on Windows: composition state, conversion state, candidate UI, and dictionaries are all IME responsibilities rather than layout responsibilities. citeturn12view2turn11search0turn11search7turn12view1

On **macOS**, Apple’s public IME framework is **InputMethodKit**. Its documentation describes classes and protocols for communication with client applications, candidate windows, and input method modes; `IMKServer` creates an input controller for each input session, and `IMKCandidates` presents candidates and notifies the controller when a user selects one. That is a textbook IME API: per-session state plus optional candidate UI. On **iOS**, the public documentation retrieved here does **not** point to InputMethodKit; instead it points to UIKit text input, `UITextInput`, and custom input views that can replace the system keyboard. So for Apple platforms the clean statement is: **IMK is the macOS IME API; iOS uses the general text-input and custom-keyboard stack rather than IMK**. citeturn10search0turn10search1turn10search5turn9view3

On **Linux**, the bottom layer is typically **XKB/XKBcommon**, which deals with keyboard model, layout, variant, and options. Above that sit input-method frameworks such as **IBus** and **Fcitx5**, which connect to applications through toolkit and protocol integration rather than replacing the underlying hardware layout system. Fcitx5’s own setup documentation shows the usual environment-variable bridge (`XMODIFIERS`, `GTK_IM_MODULE`, `QT_IM_MODULE`), and IBus input contexts accept `keyval`, `keycode`, and modifier state, then talk to an engine selected for that input context. In other words, XKB chooses what the physical keyboard means; IBus/Fcitx interpret key events and surrounding text to implement IME behaviour. citeturn12view3turn14view4turn14view5turn14view1turn14view2

The Linux IME frameworks make the composition pipeline very explicit. IBus has a **lookup table** type for candidate words and phrases and an input context API that processes key events. Fcitx5’s developer tutorial is even more direct: input methods store **state for different input contexts**, implement **candidate lists**, implement **preedit**, and handle key events. The older and adjacent ecosystems show the same pattern. The **m17n** library defines an input method object and input context, with explicit processing flow for opening an input method, creating an input context, filtering an input key, and looking up the produced text; **uim** describes itself as a multilingual input method framework; **SCIM** exposed a panel, language menu, candidate windows, and keyboard-layout mapping as part of an input-method environment. citeturn13view4turn14view1turn14view3turn13view0turn13view1turn13view2turn13view3

On **Android**, the distinction is especially sharp because the on-screen keyboard is usually the IME. Android’s documentation says an input method is implemented as a `Service`, typically by extending `InputMethodService`. IMEs may show multi-step composition by calling `setComposingText()`, and they commit final text with `commitText()`. The documentation also explicitly discusses cases where an IME “predicts text or requires multiple steps to compose a glyph or word”. Android further treats stylus handwriting as an IME capability, which confirms that handwriting belongs to the IME/modality layer rather than to the declarative layout core. citeturn15view0turn15view2turn16search0turn16search2turn16search3turn15view3

## Script families and what each method actually needs

### Chinese

Chinese is the clearest example of why a full IME exists. Microsoft’s Simplified Chinese IME exposes a **conversion candidate window** whose candidates match the typed **Pinyin** reading, and Microsoft’s Traditional Chinese IME provides corresponding IMEs for **Bopomofo**, **Changjie**, and **Quick**, all with candidate/conversion UI. Oracle’s documentation for **Wubi** says its encoding rule is based on the radical or stroke shape of Chinese characters, while Oracle’s TsangChieh documentation describes it as being based on Chinese radicals. Apple’s Stroke input source maps the five basic strokes to keys. Put plainly: Chinese methods vary in *how* the user encodes intent — pronunciation, component shape, radicals, or strokes — but the end of the pipeline is still generally **table lookup and candidate selection** over a huge character inventory. citeturn29view1turn29view2turn35search5turn34search0turn35search0turn35search3

The engineering consequence is that Chinese input is almost never just “layout plus transforms”. **Pinyin** and **Bopomofo** are fundamentally many-to-many because of homophony: one phonetic spelling corresponds to many Han characters and often many multi-character words, so a dictionary and candidate ranking are essential. **Wubi**, **Cangjie/TsangChieh**, **Quick**, and **stroke** methods are shape-based rather than phonetic, so they often reduce ambiguity, but they still rely on code tables and typically still expose candidate windows. Even Microsoft’s Traditional Chinese IME documentation describes a conversion candidate window and associated phrase suggestions; the shape-based route changes the coding scheme, not the fact that the system is an IME. citeturn29view1turn29view2turn31view5turn34search1

### Japanese

Japanese splits nicely into two stages. **Romaji → kana** is deterministic enough to model as transforms, and **direct kana input** on Japanese hardware keyboards is even closer to a plain layout. Microsoft’s Japanese IME explicitly supports switching between **Kana input** and **Romaji input**. The hard IME problem begins at **kana → kanji conversion**. Microsoft documents both a **prediction candidate window** and a **conversion candidate window**; the prediction side can use input history and the system dictionary, while the conversion side offers candidates whose reading matches what was typed. Reconversion is also supported, which is another strong signal that this is a stateful conversion engine, not a mere keyboard layout. citeturn29view0turn30view0turn30view1turn30view2turn30view3

So the Japanese boundary is precise. A layout model can cover direct kana keymaps and deterministic romaji-to-kana transforms. It should **not** attempt to embed kana-to-kanji conversion, candidate ranking, prediction, or reconversion. Those are full IME features. citeturn29view0turn30view0turn30view1

### Korean

Korean is different. The Unicode Standard defines a normative **Hangul syllable composition** algorithm from conjoining jamo, and libhangul describes itself as providing **keyboard input automata** for Korean layouts. This is exactly the kind of behaviour that sits between a layout and a full IME: it is **stateful and algorithmic**, but it is **not dictionary-driven**. The system buffers jamo, applies the composition automaton, and emits Hangul syllables. There is no inherent candidate window and no lexical search required to produce ordinary Hangul text. citeturn28view1turn28view2turn26search1

That distinction matters for model design. Korean Hangul composition needs more than a single key-to-character map, because a key sequence may merge into one syllable block, but it does **not** require the dictionary/candidate layer that Chinese and Japanese require. If a platform adds **Hanja conversion**, that is a different feature and belongs with IME-style candidate conversion, not with the core Hangul composing automaton. The ordinary Korean text-entry problem is therefore best classified as **transform/state-machine composition**, not full IME. citeturn28view1turn26search1

### Indic and other Brahmic scripts

For Indic/Brahmic scripts, **InScript** is the canonical example of a layout-level solution. C-DAC describes InScript as the standard layout for Indian-language data entry, common across Indian languages and organised around the scripts’ phonetic/alphabetic structure; Microsoft likewise describes Indic phonetic keyboards as being *in addition to* traditional **INSCRIPT** keyboards already available in Windows. That immediately suggests two buckets: **standard physical/logical keymaps** such as InScript, and **phonetic/transliteration-based input methods** that may add suggestion logic. citeturn36search1turn36search7turn29view3

The interesting complexity comes from **orthographic syllables, conjuncts, and vowel-sign ordering**. Unicode’s keyboard spec explicitly uses **reorder** rules to map typed order to desired output order and gives Devanagari examples in which `ka + halant + ka` changes ordering semantics, while a repha may sort before the base consonant and a nukta may need to follow a whole conjunct, not just a single consonant. The Unicode core specification also notes that Indic vowel signs may appear to the left, right, above, or below a consonant or consonant cluster. This is exactly the territory for a declarative transform/reorder layer: deterministic, script-grammatical, and dictionary-free. Rendering engines then shape the resulting logical sequence into conjuncts and positioned matras. citeturn19view2turn18view3turn43search7

By contrast, **phonetic Indic keyboards** can cross the boundary into IME territory. Microsoft’s Indic phonetic keyboards transliterate Latin input, **suggest possible Indic text options**, and download a **keyboard dictionary**. Deterministic transliteration without candidates can be represented as transforms; transliteration with dictionary-backed suggestions and ranking is already a lightweight IME. citeturn29view3

### Arabic and Hebrew

Arabic and Hebrew should mostly stay out of the IME bucket. For both scripts, a keyboard layout typically maps keys to base letters and, where needed, combining marks such as **harakat** or **niqqud**; dead keys or simple transforms can help with diacritics. However, **contextual shaping** in Arabic is a rendering concern, and **RTL ordering** is controlled by the bidirectional algorithm and higher-level text layout. Unicode’s core specification says Arabic shaping rules are rendering rules, while Hebrew letters are generally not cursively joined and its final forms are separately encoded. The Hebrew and Arabic layout documentation from W3C also treats directionality and layout as text-layout issues, not keyboard-layout issues. citeturn38search0turn39search0turn39search1

This means that an open layout model should not try to encode Arabic presentation forms or bidi reordering as keyboard logic. It should emit the underlying letters and marks in logical order. Optional short-vowel input is still input-layer behaviour, but shaping and directionality are not. For plain Arabic/Hebrew typing, a full IME is usually unnecessary unless the system adds transliteration, dictionary prediction, or another modality on top. citeturn38search7turn38search0turn39search1

### Southeast Asian scripts

The Southeast Asian group is mixed, but the boundary is still usable. For **Thai**, Oracle documents explicit support for **WTT** input-sequence checking, with passthrough, basic, and strict levels. That is **syntactic validation**, not lexical conversion, so it belongs naturally in a transform/validation layer rather than a full IME. Oracle’s Thai documentation also distinguishes Thai text input from Thai display/output based on WTT, which reinforces the input-vs-rendering split. citeturn40search1turn40search2turn40search0

For **Khmer** and **Lao**, the strong recurring issue in the primary sources retrieved here is the absence of spaces between words. W3C’s Khmer and Lao resources say words are not separated by spaces, spaces act more like phrase separators, and zero-width spaces may be used but are not common; CSS text specifications therefore say lexical resources are needed to identify soft wrap opportunities correctly. That behaviour matters, but it is primarily **segmentation and layout**, not key mapping. It should not be folded into a keyboard-layout core unless the user is explicitly inserting ZWSP or another delimiter. citeturn42search1turn42search2turn42search16turn42search17

For **Myanmar**, the Unicode FAQ says vowels, medials, virama, asat, anusvara, and related signs combine with letters, and it points implementers to Unicode Technical Note 11 for keyboarding and implementation issues. That is enough to classify Myanmar as needing more than one-key-one-character in many layouts: deterministic script-order handling and composition rules are required. But the material retrieved here is not sufficient to catalogue a full Myanmar reorder grammar, so the conservative conclusion is that Myanmar belongs with the **transform/reorder** bucket unless a specific engine layers dictionary or predictive conversion on top. citeturn43search5turn43search1

## LDML Keyboard 3.0 capabilities and hard limits

The Unicode **keyboard3** model can go significantly beyond key-to-character mapping. It supports **transforms**, **reorders**, **markers**, and explicit **normalisation** behaviour. A transform rewrites an input-context suffix into some other output pattern. A reorder rule assigns ordering weights, including special handling for pre-base material and tertiary ordering, so that typed sequences can be converted into stored logical order. This is specifically designed for cases like Indic orthographic syllables. citeturn19view1turn19view2turn18view3

The model also has a carefully defined **input context**, but it is intentionally limited. Transform matching operates against a temporary buffer of relevant text up to the insertion point. If the user moves the insertion point, the input context is discarded and recreated from the application text buffer; implementations may discard it at any time. That is enough for dead keys, script-order rewrites, local contextual transforms, and algorithmic composition, but it is not the same thing as a persistent IME conversion engine with dictionaries, user history, candidate ranking, or document-level language modelling. citeturn19view1

**Markers** are the keyboard model’s way to carry non-text state through local processing. The specification treats markers as opaque objects that are removed, normalised around, and then reinserted in a well-defined way so they remain associated with the intended text. Markers are removed before final output to the application, but may be retained in the implementation’s input context for later keystrokes. This makes them suitable for dead-key state, transform bookkeeping, or local script-state flags, but again it keeps the model on the declarative side of the line. citeturn18view2turn19view3

Normalisation is not incidental here; it is part of the contract. The keyboard specification says source XML and the input context are processed in **NFD** for matching and that output is normalised to the platform’s requested form, typically **NFC**. Unicode UAX #15 explains why this matters: normalisation gives canonically equivalent strings a unique binary representation, but normalised strings are not closed under concatenation, so implementations must treat boundary conditions carefully. For keyboard modelling, that means normalisation is an engine requirement around transforms and reorders, not something each layout author should reinvent. citeturn19view0turn19view3turn21view0

Where LDML deliberately stops is just as important. It defines a keyboard implementation as one that interprets the keyboard data and delivers **plain text** to the platform or application, and it separately defines IMEs as components that use contextual logic and candidate UI. There is no declarative facility here for lexical dictionaries, candidate ranking, conversion windows, phrase prediction, or handwriting/voice recognition. Those can be *referenced* by metadata, but they should not be *embedded* into the layout model itself. citeturn17view0

## Decision diagram

The cleanest engineering rule is this:

**LAYOUT** if the behaviour can be described as a fixed mapping from key position plus modifiers to a base character, string, or dead-key state, and it does not require analysing previously typed text beyond trivial dead-key context. Examples: Latin dead keys, direct kana input, InScript base maps, Arabic/Hebrew letter layouts, Thai base layouts. citeturn17view0turn12view3turn36search7

**TRANSFORMS / REORDERS** if the behaviour is deterministic, local, dictionary-free, and produces a single committed Unicode string from the user’s key sequence by rewrite, composition, normalisation, or script-order handling. Examples: romaji→kana, Hangul automata, Indic conjunct/matra ordering, Thai WTT input validation, and similar orthographic rewrites that do not need candidate ranking. citeturn19view1turn19view2turn28view1turn26search1turn40search1

**FULL IME** if the behaviour needs one-to-many candidate generation, lexical or statistical ranking, a system or user dictionary, reconversion, prediction, phrase association, or another modality such as handwriting recognition. Examples: Chinese Pinyin/Bopomofo/Wubi/Cangjie/Stroke conversion, Japanese kana→kanji conversion and reconversion, dictionary-backed Indic phonetic suggestion systems, predictive text, and handwriting modes. citeturn29view1turn29view2turn30view0turn30view1turn29view3turn14view0turn15view3

A fourth box should exist in the architecture even if it does not belong in the keyboard model: **RENDERING / LAYOUT**. Arabic contextual shaping, Hebrew/Arabic bidi behaviour, line breaking for Lao/Khmer/Thai, and glyph formation of Indic conjuncts are not keyboard logic. They are text-rendering and text-layout logic operating on the Unicode string after input. citeturn38search0turn39search0turn42search16turn42search17

## Implications for an open layout model

The model should go **far enough to cover all deterministic script behaviour**, but it should stop **before it becomes an IME framework**. Concretely, that means the model should natively represent: base maps; modifier layers; dead keys; touch layers where relevant; LDML-style transforms; reorder rules; markers; and an explicit normalisation policy or compatibility statement. That scope matches the Unicode keyboard model and captures the script-grammatical complexity needed for Hangul, Indic/Brahmic, Thai validation, and similar cases without collapsing into vendor-specific IME engines. citeturn17view0turn19view0turn19view1turn19view2

The model should *not* embed full IMEs. Instead, it should provide **declarative references** to them. LDML already has a useful naming precedent in the BCP 47 transformed-content extension data: `k0` is defined for **keyboard transforms** and `i0` for **input method engine transforms**. An open layout model can reuse that spirit by identifying transform sets and external IME dependencies in standardised metadata rather than by trying to serialise dictionaries and candidate engines into the layout itself. citeturn17view1

In practice, the metadata should include at least these declarations:

- a stable **layout identifier** and locale/script coverage;
- the **physical-key assumption** or geometry family where relevant;
- the **normalisation contract** for matching and output;
- any imported **transform/reorder set identifiers**;
- marker definitions and any non-default backspace semantics;
- capability flags such as **needs surrounding text**, **uses algorithmic composition**, or **requires reorder rules**;
- and, crucially, **external IME references** by platform, such as TSF text service identifiers on Windows, InputMethodKit/input-source identifiers on macOS, Android IME package/service identifiers, or IBus/Fcitx engine names on Linux. This lets a layout say “ships transform set Y” or “requires IME X” while staying interoperable with LDML-style keyboard data. citeturn12view2turn10search0turn15view2turn14view2turn14view4turn17view1

Adjacent features should be modelled as **separate optional layers**, not as the layout core. Predictive text and autocorrect belong with dictionaries and user history. Microsoft’s Japanese IME prediction window explicitly uses input history and the system dictionary; Microsoft’s Indic phonetic keyboards download a dictionary; Android and IBus expose handwriting-related IME capabilities. Transliteration input is split: purely deterministic transliteration can be a transform set, but transliteration with suggestions and ranking is an IME feature. Voice input is further still from layout modelling, because it is not driven by key events at all. citeturn30view1turn29view3turn14view0turn15view3

### Open questions and limitations

The broad boundary is high-confidence, but two low-level areas remain intentionally conservative in this report. First, the exact reorder grammars for **Khmer** and especially **Myanmar** vary by engine and were not exhaustively enumerated from the primary material retrieved here; the report therefore classifies them at the family level as needing deterministic script-order handling without claiming a complete rule inventory. Second, on Apple platforms the retrieved documentation clearly supports **InputMethodKit on macOS** and **custom text-input/custom-keyboard APIs on iOS**, but it does not provide a single Apple page spelling out that distinction as a comparative architecture document, so that part of the report is phrased carefully and narrowly. citeturn42search1turn42search2turn43search5turn10search0turn9view3

# Physical Keyboard Geometry Formats for an Open Layout Model

## Executive summary

For engineers designing an open keyboard layout model, the existing ecosystem falls into three distinct layers. **KLE JSON** is the de-facto hobbyist interchange for *drawing* key geometry, with strong support for key sizes, offsets, two-rectangle odd shapes, rotation clusters, and rich legend placement, but no electrical matrix and no durable physical-key identifiers. **QMK `info.json`** turns that visual description into a firmware-facing schema by making each key stateless and adding matrix coordinates, but it drops true odd-shape support and, as of the current documentation, still treats rotation as “currently not implemented” in downstream renderers. **VIA and Vial** build on KLE-style geometry, but they encode matrix positions, layout options, and encoder metadata into legend fields, which is practical for existing tooling and awkward as a clean long-term schema. **XKB geometry** is the opposite: it is expressive and standardised in the X world, with explicit shapes, sections and mm/10 units, but it is legacy, complex, and now only “parsing only” in libxkbcommon because the format saw little real-world use. **ISO/IEC 9995-1** is not a full geometry format at all: it is the stable reference grid and naming system that underlies row/column position identities and should be treated as the backbone for durable physical-key IDs. citeturn45view0turn16view1turn17view1turn17view2turn33view2turn34view0turn24view1turn20view0

The practical recommendation is therefore to make the open model’s **geometry layer stateless and absolute**, with explicit `x`, `y`, `w`, `h`, optional rotation pivot and angle, optional composite/polygonal shapes, and stable **physical-position identifiers** derived from the ISO/XKB/UI/HID family rather than from legends or matrix addresses. Matrix coordinates, layout options, legends, and character mappings should live in separate layers or sidecars. That choice maps cleanly outward: it exports well to QMK, can be serialised to KLE with a row-wise stateful encoder, can drive VIA/Vial option groups, and can annotate keys with UI Events `code`, XKB names, and USB HID usages without conflating any of those with the physical geometry itself. citeturn16view1turn17view1turn18search0turn33view0turn20view0turn39view0

## What the existing formats are really modelling

The first thing to keep clean is the distinction between **physical geometry** and **character mapping**. ISO/IEC 9995-1 describes keyboard sections, their relative placement, and a key-numbering system for numeric, alphanumeric and composite keyboards; it is explicitly about the physical organisation and reference system, not the language mapping of characters to keys. W3C’s UI Events `code` specification then adopts that same physical-key idea for the web: `code` values are based only on the key’s physical location and do not vary with locale. The canonical example is the physical key at the Q-position: in the spec it is `KeyQ`, even though on French AZERTY that same physical key is labelled `A`. citeturn24view1turn30view3turn20view0turn21view0

That clean separation is not preserved equally well by actual tooling. KLE is primarily a *visual* editor and serialisation format. QMK adds *firmware* concerns such as the scan matrix and `LAYOUT_*` macros. VIA and Vial go further by mixing *configuration UI state* into the legend channels of KLE data. XKB splits things into separate components—`keycodes`, `symbols`, `types`, `compat`, and optional `geometry`—which is conceptually cleaner, but the geometry component has largely withered in practice. Ergogen and KiCad tooling show another pattern again: they treat geometry as a set of points and transforms that later drive plates, outlines and PCB footprints, then bridge outward to KLE/QMK/VIA formats when needed. citeturn45view0turn16view1turn17view0turn17view1turn17view2turn34view1turn37view0turn37view1turn37view2

For an open model, the implication is straightforward: **the geometry layer should be authored as geometry**, not as a disguised keymap, not as a firmware macro, and not as UI text placed on the keycaps. The existing ecosystem rewards that choice because almost every practical export path is easier from a stateless absolute geometry core than from highly coupled row-wise or legend-encoded formats. citeturn16view1turn17view1turn17view2turn37view2

## KLE JSON as the de-facto geometry interchange

KLE’s serialised format is a top-level JSON array containing optional keyboard metadata followed by arrays of keys, one array per “row”. Within a row, plain string entries are keycaps, and property objects inserted before a key modify either just the next key or all subsequent keys. The row-based serialisation is deliberately compact: the first row starts at `y = 0`, each subsequent row increments `y` by `1`, each row resets `x = 0`, and after each key the current `x` is advanced by the previous key’s width. That means KLE is not fundamentally an absolute per-key data model; it is a **stateful cursor language** for laying out keys in sequence. citeturn45view0turn10view0

KLE’s geometry primitives are nevertheless very capable for the keyboard hobbyist ecosystem. The core rectangle is given by `x`, `y`, `w`, `h` in keyboard units where `1u` is a standard 1×1 key. For oddly shaped keys, KLE provides `x2`, `y2`, `w2`, `h2`, defining a second rectangle relative to the primary one. In the KLE parsing library, the deserialised key exposes these as `x`, `y`, `width`, `height`, `x2`, `y2`, `width2`, `height2`, together with `rotation_x`, `rotation_y`, and `rotation_angle`. The second rectangle is how KLE represents stepped keys and the visual outline of keys such as ISO Enter; labels, however, remain positioned relative to the main rectangle, not the unioned outline. citeturn45view0turn10view0

Rotation is where KLE’s hidden quirks matter most for an exportable open model. In the parser used by `kle-serial`, serialized rotation properties are `r`, `rx`, and `ry`, but they are only accepted on the **first key in a row**; otherwise deserialisation throws an error saying rotation can only be specified on the first key in a row. At the end of each row, the parser also resets `current.x` to `current.rotation_x`, which is why rotated clusters behave like row groups anchored to a shared centre of rotation rather than like independently rotatable absolute keys. KLE’s own release notes describe this as support for “rotated key clusters” whose keys share an angle and centre-of-rotation. This is one of the strongest arguments *against* making row order or cluster state the canonical geometry in a new model: those are KLE serialisation constraints, not intrinsic facts about the physical keyboard. citeturn14view1turn14view2turn14view3turn11view0

KLE also carries a surprisingly rich legend-rendering model. A key can have up to 12 legends, separated by newline characters in the key string; alignment is controlled by the bitfield `a`, and the KLE wiki notes that, due to historical evolution, the default alignment is `4`, not `0`. The format supports primary and secondary font sizes (`f`, `f2`), text colour (`t`), centred and side-printed legends, and profile hints (`p`) such as DCS, DSA, SA, plus row modifiers like `R1`–`R5` and `SPACE`. The README of `kle-serial` further exposes `labels`, `textColor`, `textSize`, `default.textColor`, `default.textSize`, as well as `nub`, `stepped`, `decal`, `ghost`, and `profile`. This makes KLE valuable for *rendering metadata*, but it also shows why legends should not be the physical identity of a key: they are presentation-centric and may not even reflect the eventual firmware mapping. citeturn45view0turn10view0

KLE’s limitations are equally important. It does **not** encode the electrical matrix. It has **no built-in stable physical key identifier** beyond row order and text labels. The “raw data” editor is not strict JSON—it relaxes quoting rules and omits the outer brackets—while downloaded/uploaded JSON is strict, which is a real-world interoperability trap. Odd shapes are limited to the union of two rectangles, not arbitrary polygons. Variant handling is mostly social convention: people make separate KLE files, use decals, or place alternative keys off to the side, but KLE itself has no first-class concept of “these two keys are mutually exclusive build options”. All of this makes KLE an excellent **import/export target**, but a poor choice for the authoring truth of an open model. citeturn45view0turn10view0turn11view0

## QMK, VIA and Vial

### QMK `info.json` and `LAYOUT_*` macros

QMK’s `info.json` improves on KLE by normalising the data into **stateless per-key records**. In the current reference, each entry of a `layouts.LAYOUT_<name>.layout` array carries a required `matrix` coordinate and absolute `x`/`y` positions in key units, plus optional `w`, `h`, `label`, `hand`, and even `r`, `rx`, `ry`. The same page explicitly says that, unlike KLE, each key is stateless and inherits no properties from previous keys, and that all positions and rotations are specified relative to the top-left corner of the keyboard and the top-left corner of each key. That statelessness is exactly the right direction for an open geometry schema. citeturn16view1

The connection to firmware happens through the `LAYOUT_*` C macro. QMK’s Configurator support guide shows a conventional macro whose positional arguments (`k00`, `k01`, …) are then assembled into a two-dimensional matrix array with explicit holes. The same guide explains that Configurator ties each visual object in the JSON to the corresponding key *in sequence* from the layout macro. In other words, QMK still depends on a positional ordering contract, but it augments that with the actual scan-matrix address and a stateless geometric record. The names `k00`, `k01`, and so on do not matter semantically; what matters is consistency between the macro’s formal parameters, the macro body, and the JSON order. citeturn15view1turn16view0

QMK also has explicit conventions for multiple physical variants. The hardware keyboard guidelines require multiple layouts to be expressed as several `LAYOUT_*` definitions and recommend a superset `LAYOUT_all` when a PCB supports more switch positions than any single physical build can use. The examples given include `LAYOUT_all`, `LAYOUT_ansi`, and `LAYOUT_iso`, with matching `default`, `default_ansi`, and `default_iso` keymaps. That is a useful export target, but it also reveals a modelling truth: PCB realities and physical build realities are not identical, so a clean open model should not collapse them into one array. citeturn15view2

QMK’s limitations on pure geometry are substantial. The reference still documents `r`, `rx`, and `ry` as **currently not implemented**. The Configurator support page says the Configurator does not support key rotation or non-rectangular key shapes such as ISO Enter, and that contributors must manually work around vertically offset keys that confuse the KLE-to-JSON converter. QMK’s “solution” for ISO Enter is a convention: represent it as a `1.25u × 2u` rectangle aligned to the right edge of the alpha block. In other words, QMK geometry is best thought of as a **rectangular UI-placement layer with matrix bindings**, not as a complete physical shape description. citeturn16view1turn16view0

### VIA and Vial keyboard definitions

VIA’s specification says a keyboard definition JSON describes the physical layout of keys, layout options, encoders and other configurable elements, and that the definition is keyed to the device’s VID/PID and matrix dimensions. The crucial representational choice, though, is in the VIA Layouts documentation: a VIA keyboard layout is based on **KLE JSON**, and the matrix position of each key is expected in the **top-left legend**. Layout options are then encoded in the **bottom-right legend** as `option,choice`, while an optional `labels` array names those options for the UI. This is clever and practical, but it means geometry, matrix identity and variant semantics are all being tunnelled through presentation legends. citeturn17view0turn17view1

VIA’s option model is stronger than KLE’s. The documentation requires layout-option choices to have the same coverage, to overlap exactly in shape, and to be aligned either vertically or horizontally with the default choice. It recommends defining options at the finest practical granularity: not one monolithic “ANSI vs ISO” switch, but separate choices such as ISO Enter, split left Shift, split right Shift, and bottom-row variants. Decal keys may be used for blockers such as HHKB or WKL layouts. This is valuable prior art for an open model: variant choice should be a first-class object with an explicit coverage region and multiple mutually exclusive key sets. citeturn17view1

VIA also treats some non-character controls as pseudo-keys in the layout view. Encoders without push switches are represented by KLE “keys” whose centre legends are `e0`, `e1`, and so on; encoders with push switches combine that with a normal matrix-bound key. The QMK configuration guidance for VIA adds one more important firmware-side constraint: the `via` keymap should use a `LAYOUT_*` macro that exposes **all electrical positions**, even if the resulting physical arrangement is impossible. That again argues for separate geometry and electrical layers in the open model. citeturn17view1turn17view3

Vial is close to VIA in spirit, but the data contract is not identical. Vial’s porting guide says `vial.json` is a fixed-format JSON file describing the physical layout for the Vial GUI, that it is based on KLE data with added information encoded within legends, and that its structure is “vastly different” from QMK’s `info.json`. It uses the top-left legend as `row,col` for matrix identity and uses bottom-right KLE legends for options in the same general style as VIA. The guide also notes that `name`, `vendorId`, and `productId` are required by VIA but *not needed* by Vial. For encoders, Vial goes further still and represents an encoder as two or three 1u pseudo-keys, with the centre legend `e` and the top legend encoding encoder index and action, explicitly separate from the switch matrix. citeturn17view2turn18search0turn18search1

The upshot is that VIA and Vial are extremely important export targets, but they should be treated as **projection formats**. Their legend-encoding tricks are optimised for current configurator UIs, not for clean long-term physical modelling. citeturn17view1turn17view2turn18search0

## Standard references and stable physical-key identity

ISO/IEC 9995-1 supplies the durable reference frame. The published ISO abstract says the standard identifies keyboard sections, specifies their general shape and relative placement, and defines a key-numbering system for numeric, alphanumeric and composite keyboards. W3C’s UI Events `code` specification explicitly says its row names—from `A` at the bottom of the alphanumeric section to `E` at the top, with the function row as `K`—are consistent with ISO/IEC 9995-1. This is the right foundation for an open geometry model because it yields locale-neutral *positions* rather than locale-dependent symbols. citeturn24view1turn20view0

The UI Events spec is also the clearest concise source for standard physical form factors. It distinguishes the standard “101” layout, the European “102” layout, the Korean “103”, Brazilian “104”, and Japanese “106”. The 102 layout adds `IntlBackslash` next to the left Shift and a second key tucked under Enter, encoded as `Backslash`. The Brazilian 104 layout adds the two 102-layout keys plus `IntlRo` and an extra numpad key `NumpadComma`. The Japanese 106 layout adds `IntlYen`, `Backslash`, `IntlRo`, and three input-mode keys—`NonConvert`, `Convert`, and `KanaMode`—by shortening the spacebar. These distinctions are more useful as **variant sets over a shared geometry** than as whole separate keyboard “types”. citeturn21view2turn21view3turn21view4

XKB contributes the other half of the identity story: the naming scheme used across Linux keyboard stacks. libxkbcommon’s documentation says key names like `<AE01>`, `<AD01>`, and `<AC01>` are based on ISO/IEC 9995-1 and denote positions in the keyboard grid; its example explains `<AE01>` as the alphanumeric section, row `E`, column `01`. XKB keycodes can then map those names to platform key numbers, while symbolic names remain stable across layouts. That is very close to what an open physical-geometry model wants from a stable key-position name. citeturn33view0

A worked example shows how the same physical key travels through the stack. The physical key in row `D`, column `01` of the alphanumeric block is the **Q-position** on US QWERTY. In XKB naming it is `<AD01>`. In UI Events it is `KeyQ`, and the W3C spec explicitly notes that on French AZERTY the same physical key is labelled `a`. In USB HID, the corresponding keyboard-page usage is `0x14` “Keyboard q and Q”. KLE, QMK, VIA and Vial, however, do **not** provide a universal position identity for that key: KLE only gives you layout order and whatever legend text you wrote; QMK gives you a matrix coordinate and array order; VIA/Vial typically hide the matrix coordinate in the legend. The conclusion is unavoidable: the open model must define its own stable physical key ID and then attach ISO/XKB/UI/HID annotations to it, not the other way round. citeturn33view0turn21view0turn44view0turn45view0turn16view1turn17view1turn17view2

This reconciliation also exposes real-world disagreements. For Brazilian layouts, UI Events says the extra key next to right Shift is `IntlRo`, but USB HID 1.7 footnotes say the Brazilian `/ ?` key should instead use `Keyboard International1` and even suggest renaming that usage because it is no longer uniquely Japanese. That kind of mismatch is exactly why the physical model should carry **cross-references** rather than using any single downstream code system as the canonical identity. citeturn21view2turn21view4turn44view3

Non-character keys belong in the same physical model. UI Events defines separate location-neutral codes for numpad keys such as `Numpad0` through `Numpad9` and `NumpadEnter`, for function keys `F1`–`F12` and beyond, for host-control/media keys such as `MediaPlayPause`, `MediaTrackNext`, `AudioVolumeUp`, and for special keys such as `Fn` and `FnLock`. USB HID likewise distinguishes keyboard-page usages for ordinary typing and many keyboard-like function keys, while media transport controls such as Play/Pause and Scan Next Track live on the Consumer page. Importantly, the UI Events spec notes that `Fn` is usually a hardware key that does **not** generate a separate code. A geometry model must therefore be able to place keys that are physically present but may have no OS-visible event identity. citeturn21view4turn43view0turn43view1turn43view2turn43view3turn40view3turn40view4turn42view0turn42view2turn42view3

## Implications for an open layout model

The model should make **absolute per-key geometry** the source of truth. In practice that means `x`, `y`, `w`, and `h` in abstract keyboard units, with the keyboard origin at the top-left of the unrotated overall layout, mirroring QMK and XKB rather than KLE’s row cursor semantics. Store `row` and `column` only as optional organisational metadata or reference-grid annotations, not as authoritative geometry: split, columnar, staggered, rotated and option-dependent boards do not reduce cleanly to a single row/column lattice, and KLE’s row behaviour is a serialisation trick rather than a fundamental physical property. citeturn16view1turn33view2turn45view0turn37view0

Rotation should be explicit and absolute. The minimum interoperable representation is a per-key or per-cluster `rotation.angle`, `rotation.pivot_x`, and `rotation.pivot_y`, in degrees for easier mapping to KLE/QMK/XKB. A purely per-key angle without a pivot loses information needed by KLE and XKB. Conversely, the canonical model must *not* inherit KLE’s “rotation only on first key in row” constraint; that should exist only in the KLE exporter. citeturn14view1turn14view2turn33view2

For shapes, the model should go a little beyond the lowest common denominator. A good compromise is: rectangle by default; optional list of additional rectangles for KLE-style composite shapes; and optional generic polygon/outline support for higher-fidelity consumers such as SVG/CAD or XKB-style exports. The exporter can then degrade gracefully: KLE gets one or two rectangles; QMK gets a bounding-box rectangle or a format-specific proxy; VIA/Vial inherit the KLE-compatible shape view; XKB can use full outlines if needed. That is more future-proof than hard-wiring the model to KLE’s two-rectangle limit or QMK’s rectangle-only UI. citeturn10view0turn16view0turn33view1turn33view2

The model should have a **mandatory stable physical key ID** that is neither a legend nor a matrix coordinate. For standard positions, the best practice is to attach structured cross-references such as `iso_grid`, `xkb_name`, `ui_code`, and `usb_hid`, while keeping a neutral internal `id` as the authoritative handle. A key at the Q-position, for example, might carry `iso_grid: "D01"`, `xkb_name: "AD01"`, `ui_code: "KeyQ"`, and `usb_hid: { page: 0x07, usage: 0x14 }`. The important point is that these are *annotations*, not the primary key of the schema, because some special keys do not map neatly across all systems and some downstream systems disagree on naming, as the Brazilian extra key demonstrates. citeturn33view0turn21view0turn44view0turn21view2turn44view3

Matrix data should be kept adjacent but separate. A practical design is a sidecar relation such as `electrical_bindings`, keyed by physical key ID, that can hold one or more matrix addresses, firmware macro positions, encoder indices, or “shared footprint” facts. That cleanly handles QMK’s `matrix` requirement, VIA/Vial’s top-left `row,col` legend convention, and the reality that `LAYOUT_all` or optional footprints may expose electrical positions that are not simultaneously buildable. Geometry remains geometry; firmware binding remains firmware binding. citeturn16view1turn15view2turn17view1turn17view2turn17view3

Variants should be first-class objects, not separate full layouts unless necessary. The model should represent each build-dependent region as a **variant group** with mutually exclusive choices and an explicit coverage region. That mirrors what VIA/Vial already require for layout options and is a much better fit for ANSI/ISO/JIS/ABNT differences than duplicating the entire keyboard. In practice, that means separate variant groups for at least the Enter cluster, left/right Shift clusters, bottom row, split/stepped Backspace, split/stepped Space, blockers, and JIS/ABNT language-specific additions. The exporter can then collapse those groups into VIA/Vial `option,choice` legends, emit multiple QMK `LAYOUT_*` arrays, or generate separate KLE views if a downstream tool has no native option semantics. citeturn17view1turn18search0turn15view2

Non-character keys should be placed exactly like character keys. The geometry layer must represent function keys, arrow/navigation keys, numeric keypad keys, media keys, launch/application keys, encoders, and physically present hardware-layer keys such as `Fn`. What differs is not the geometry but the identity annotations and host visibility. A useful pattern is a `presence` or `eventing` field, for example `host_visible: true/false`, plus optional `ui_code` and `usb_hid` annotations when a host-facing identity exists. `Fn` is the canonical case for why this matters: it is physically real, often central to laptop ergonomics, and yet often generates no separate OS-visible code. citeturn43view0turn43view1turn40view4turn42view0turn42view2turn42view3

Legend placement should remain **outside the core geometry** and live in a rendering sub-layer. KLE proves that people care about 12-slot legend placement, front legends, size and profile hints. Those are worth preserving as optional render metadata for round-tripping to KLE or for generating documentation images. They should not, however, be mandatory to describe where a physical key is. Likewise, keycap profile data and homing indicators are best stored as optional rendering/manufacturing hints, not as geometry-defining facts. citeturn45view0turn10view0

A minimal hand-maintainable record, therefore, would look conceptually like this:

## Open questions and limitations

The main limitation of the source material is that the full ISO/IEC 9995-1 text is paywalled. This report therefore relies on the ISO catalogue abstract and preview pages, plus standards that explicitly build on ISO 9995-1—especially W3C UI Events and XKB/libxkbcommon—to recover the practical row/column naming and position-reference scheme. citeturn24view1turn30view3turn20view0turn33view0

The second limitation is that XKB’s special-case historical key names outside the neat `<AE01>/<AD01>/<AC01>` pattern were not fully documented in the sources reviewed here. That does not affect the main recommendation—use your own stable IDs and attach XKB names as annotations—but it does mean an implementation should expect a small mapping table for legacy names when importing XKB data. citeturn33view0turn34view1

Finally, some details are genuinely fast-moving. In particular, QMK’s geometry support can evolve, and the current docs still say rotation is “currently not implemented”; VIA/Vial definition conventions also evolve with their configurators. Those format-specific edges should therefore be implemented as versioned exporters/importers around a stable internal geometry schema, not baked into the schema itself.
